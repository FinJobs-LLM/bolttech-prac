# MLflow in this repository

MLflow is the **experiment-tracking and audit layer**. Its job is to make the model-selection
process *transparent and reproducible*: every model trained тАФ baselines, optimized models, and every
single Optuna trial тАФ is recorded with its hyperparameters, imbalance strategy, metrics, plots and
the serialized model, so anyone can answer "which model won, with what settings, and why" without
re-running the pipeline.

It does **not** train, optimize, or select models. That is done by scikit-learn / the boosting
libraries (training), [Optuna](./optuna.md) (search), and `run_pipeline.py` (selection). MLflow only
*observes and stores*.

## Where it lives in the code

| File | Role |
|---|---|
| `src/ml/mlflow_tracking.py` | Initialization + small logging helpers. |
| `src/ml/optimize_optuna.py` | Opens the per-family **parent** run and one **nested child run per trial**. |
| `src/run_pipeline.py` (`evaluate_model`) | Opens one **evaluation run per model** (baseline & optimized) and logs metrics, figures and the model artifact. |

## Configuration

Set once in `src/ml/mlflow_tracking.py::init_mlflow()`:

- **Tracking store:** a local file store at `./mlruns` (`mlflow.set_tracking_uri(<repo>/mlruns)`).
  No database or server process is required тАФ the project stays self-contained.
- **File-store opt-in:** MLflow 3.x treats the filesystem backend as "maintenance mode" and errors
  unless `MLFLOW_ALLOW_FILE_STORE=true` is set; `init_mlflow()` sets it automatically.
- **Experiment name:** `claim_approval_optimization` (`config.MLFLOW_EXPERIMENT`).

`mlruns/` is git-ignored тАФ it is regenerated by running the pipeline.

## Run hierarchy

Running `python src/run_pipeline.py` produces (with `N_TRIALS = 50`) roughly **210+ runs** in the
experiment, organized as:

```
experiment: claim_approval_optimization
тФЬтФАтФА RandomForest_baseline        (evaluation run)
тФЬтФАтФА XGBoost_baseline             (evaluation run)
тФЬтФАтФА LightGBM_baseline            (evaluation run)
тФЬтФАтФА CatBoost_baseline            (evaluation run)
тФЬтФАтФА RandomForest_optuna          (parent: stage=optuna_search)
тФВ   тФЬтФАтФА RandomForest_trial_0     (nested: stage=optuna_trial)
тФВ   тФЬтФАтФА RandomForest_trial_1
тФВ   тФФтФАтФА тАж 50 trial runs, each logging its params + val_pr_auc
тФЬтФАтФА XGBoost_optuna   тФЬтФАтФА тАж 50 trials
тФЬтФАтФА LightGBM_optuna  тФЬтФАтФА тАж 50 trials
тФЬтФАтФА CatBoost_optuna  тФЬтФАтФА тАж 50 trials
тФЬтФАтФА RandomForest_optimized       (evaluation run)
тФЬтФАтФА XGBoost_optimized            (evaluation run)
тФЬтФАтФА LightGBM_optimized           (evaluation run)
тФФтФАтФА CatBoost_optimized           (evaluation run)
```

This is the nested structure the brief asks for: **parent run = model family, child runs =
individual Optuna trials.**

## What is logged

### Optuna parent run (`<Model>_optuna`)
- Tags: `stage=optuna_search`, `model=<family>`.
- Params: `best_<param>` for every hyperparameter of the best trial.
- Metric: `best_val_pr_auc` (the best validation PR-AUC found by the search).

### Optuna trial run (`<Model>_trial_<n>`, nested)
- Tags: `stage=optuna_trial`, `model=<family>`.
- Params: the exact hyperparameters sampled for that trial.
- Metric: `val_pr_auc` for that trial.

These trial runs are what let the **MLflow / Optuna** dashboard page (and the MLflow UI) show how the
score evolved across the search.

### Evaluation runs (`<Model>_baseline`, `<Model>_optimized`)
Logged in `run_pipeline.py::evaluate_model`:
- **Tags:** `model`, `stage` (`baseline`/`optimized`), `imbalance_strategy`, `preprocessing`
  (`native categorical` for CatBoost, else `median impute + one-hot`).
- **Params:** `hp_<name>` for each hyperparameter, `threshold` (tuned on validation),
  `imbalance_strategy`.
- **Class distribution:** `class_count_Completed`, `class_count_Declined`.
- **Metrics:** every value from `evaluate.compute_metrics`, prefixed `val_*` and `test_*` тАФ including
  `pr_auc`, `roc_auc`, `f1_declined`, `recall_declined`, `precision_declined`, `balanced_accuracy`,
  `accuracy`, `false_positives`, `false_negatives` (the confusion-matrix nested structure is skipped).
- **Artifacts (`figures/`):** confusion matrix, Precision-Recall curve, ROC curve, and feature
  importance (the PNGs in `reports/figures/`).
- **Model artifact (`model/claim_model.joblib`):** the full `ClaimModel` (preprocessor + estimator +
  threshold), so any run's exact model can be reloaded.

## How it answers the brief's questions

The MLflow UI lets a reader understand:
- **Which model performed best** тАФ sort evaluation runs by `val_pr_auc` / `test_pr_auc`.
- **Which hyperparameters were selected** тАФ the `hp_*` params on the `*_optimized` run and `best_*`
  on the `*_optuna` parent.
- **Why it beats the alternatives** тАФ compare `test_pr_auc`, `test_recall_declined`, etc. side by side.
- **How the imbalance strategy mattered** тАФ the `imbalance_strategy` tag plus baseline-vs-optimized
  metrics.
- **Whether it detects the minority `Declined` class** тАФ `*_recall_declined`, `*_pr_auc`, the
  confusion-matrix image.

## Viewing the runs

```bash
mlflow ui --backend-store-uri ./mlruns --port 5000
# open http://localhost:5000  (experiment: claim_approval_optimization)
```

> **Model Registry note.** The file store does not support the MLflow Model Registry
> (`mlflow.register_model` needs a database backend, e.g. `sqlite:///mlflow.db`). Instead, the
> selected best model is saved directly to `models/best_model.joblib` (+ `best_model_meta.json`) and
> is also attached as an artifact to its evaluation run.

## Relationship to the rest of the pipeline

```
Optuna searches тФАтФАтФР
                  тФЬтФАтЦ║ every trial & model тФАтФАтЦ║ MLflow (records params/metrics/artifacts)
training/eval тФАтФАтФАтФАтФШ
run_pipeline selects best by validation PR-AUC тФАтФАтЦ║ models/best_model.joblib тФАтФАтЦ║ FastAPI /predict
```

See [optuna.md](./optuna.md) for the search side.
